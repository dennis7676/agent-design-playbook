# Chapter 5: 멀티 에이전트 운영 — 에이전트 하나로 부족할 때

> 에이전트 하나를 관리하는 건 설계 문제입니다. 셋을 관리하는 건 **조직** 문제입니다.

## TL;DR

에이전트를 여러 개 돌리면 확장이 될 것 같습니다. 현실에서는 단일 에이전트에서 존재하지 않던 세 가지 협업 문제가 터집니다: 소유권 모호, 맥락 단절, 인수인계 실패. 이 챕터는 3-에이전트 시스템이 "에이전트가 많으면 더 많이 해내겠지"에서 "에이전트가 많으니 더 혼란스럽다"를 거쳐, 다시 생산적인 상태로 돌아온 과정을 추적합니다. 핵심은 누가 뭘 맡고, 상태를 어떻게 공유하며, 이음새에서 무슨 일이 벌어지는지를 명시적 프로토콜로 잡아낸 것입니다.

---

## 팀 확장의 착각

에이전트를 추가하는 건 달콤한 유혹입니다. 에이전트 하나가 볼트를 잘 관리하고 있으면, 자동화 파이프라인 전담 에이전트를 하나 더 붙이면 어떨까? 실시간 대화 담당도 하나?

저도 그렇게 했습니다. 에이전트 셋, 각자 명확한 전문 영역을 가진 구성입니다:

| 에이전트 | 역할 | 환경 |
|---------|------|------|
| **Eve** | 두뇌 — 설계, 추론, 보안 민감 작업 | Mac Mini (24/7) |
| **Chloe** | 손 — 자동화 파이프라인, 데이터 수집 | n8n (Docker) |
| **Claude** | 빌더 — 볼트 관리, 복잡한 추론, 콘텐츠 | Claude Code (인터랙티브) |

종이 위에서는 깔끔한 분업입니다. 실전에서 첫 한 달은 혼돈이었습니다.

---

## 세 가지 협업 문제

### 문제 1: "이거 누구 건데?"

가장 먼저 깨진 건 소유권이었습니다. 회의 녹취록이 들어왔을 때 누가 처리하죠? Eve가 추론할 수 있고, Chloe의 파이프라인이 라우팅할 수 있고, Claude가 볼트 맥락을 더해 보강할 수 있습니다.

소유권이 명확하지 않으면, 셋 모두가 같은 항목을 부분적으로 처리합니다 — 아니면 더 심하게, 아무도 안 합니다. 다른 에이전트가 했겠거니 하면서.

조직으로 치면 **책임 분산(diffusion of responsibility)** 문제 그 자체입니다. 역할이 불명확한 팀에서는 급한 일은 세 번 처리되고, 중요한 일은 한 번도 처리되지 않습니다.

**해결: 명시적 라우팅 규칙.**

정교한 협업 프로토콜이 필요한 게 아니었습니다. 단순한 라우팅 테이블이면 충분했습니다:

| 입력 유형 | 담당 | 지원 |
|----------|------|------|
| 수신 메시지 (Telegram) | Eve (첫 응답) | Claude (에스컬레이션) |
| 스케줄 파이프라인 (cron) | Chloe (실행) | Eve (에러 처리) |
| 인터랙티브 세션 (터미널) | Claude (주 담당) | Eve (비동기 리뷰) |
| 인박스 분류 | Claude (/inbox 커맨드) | Chloe (전처리) |

여기서 핵심 통찰이 나옵니다: **소유권은 역량이 아니라 환경에 따른다.** Eve가 실시간을 담당하는 이유는 24/7 돌아가니까. Chloe가 파이프라인을 담당하는 이유는 n8n에 살고 있으니까. Claude가 인터랙티브 작업을 담당하는 이유는 터미널에서 실행되니까. 라우팅은 스킬이 아니라 인프라를 따릅니다.

### 문제 2: "방금 무슨 일이 있었지?"

두 번째 문제는 맥락 단절이었습니다. 각 에이전트가 자기만의 세션, 자기만의 컨텍스트 윈도우에서 움직였습니다. Eve가 새벽 2시에 내린 결정을, 사용자가 아침 세션을 시작할 때 Claude는 모릅니다.

증상이 미칠 지경이었습니다:
- Claude가 Eve가 밤새 이미 끝낸 작업을 새로 제안함
- Chloe의 파이프라인이 Claude가 수동으로 처리한 항목을 다시 처리함
- Eve가 Claude가 수정한 걸 모르고 예전 버전의 맥락으로 문서를 리뷰함

조직으로 치면: 이메일 참조를 절대 안 하는 직원 셋입니다.

**해결: 맥락 공유가 아니라 상태 공유.**

컨텍스트 윈도우를 합칠 수는 없습니다 — 에이전트마다 고유합니다. 대신 세 가지 공유 상태 메커니즘을 만들었습니다:

**1. Future Board (공유 태스크 트래커)**

모든 에이전트가 읽고 쓰는 마크다운 파일입니다. 모든 태스크에 담당자, 상태, 인수인계 이력이 있습니다:

```markdown
| FUTURE-20260322-001 | Dashboard v2 | Claude | WIP | 2026-03-22 |
  Handoff: "2026-03-22 [Claude→Eve]: Phase 1 done, review layout"
  Handoff: "2026-03-22 [Eve→Claude]: Layout approved, proceed to Phase 2"
```

어떤 에이전트든 세션을 시작하면 먼저 Future Board를 읽습니다. "그거 네가 하고 있었어?"라는 상황을 원천 차단합니다.

**2. 대화 로그 (에이전트별 JSONL)**

각 에이전트가 매 세션마다 구조화된 로그를 남깁니다:

```jsonl
{"ts":"2026-03-26T02:15:00+09:00","agent":"eve","role":"system","content":"session_start"}
{"ts":"2026-03-26T02:16:00+09:00","agent":"eve","role":"assistant","content":"Processed 5 inbox items, routed 3 to projects"}
```

Claude가 아침 세션을 시작하면, `/start`가 Eve의 야간 로그에서 관련 변경사항을 스캔합니다. 전체 대화가 아닙니다 — 구조화된 요약만.

**3. CLAUDE.md — 공유 헌법**

세 에이전트 모두 같은 CLAUDE.md를 참조합니다. 같은 판단 프레임워크로 결정하고, 같은 위험 게이트를 따르고, 같은 자율성 경계를 존중하도록 보장합니다. 이것 없이는 각 에이전트가 "이걸 자율적으로 해도 안전한가?"에 대해 제각각의 해석을 발전시킵니다.

### 문제 3: "인수인계가 빠졌다"

세 번째 문제가 가장 교묘했습니다: 인수인계 실패. 에이전트 A가 작업을 시작하고, 중단 지점에 도달하고, 에이전트 B가 이어받아야 합니다. 둘 사이에 전달되는 정보가 매끄러운 연속이 될지, 처음부터 다시 시작이 될지를 결정합니다.

**실패 양상**: 에이전트 A가 "태스크 거의 다 됐고, 마무리만 하면 됩니다"라고 씁니다. 에이전트 B가 이걸 읽고, "거의 다 됐다"가 뭔 뜻인지 알 수 없어서 작업의 60%를 다시 하고, A가 이미 완료한 부분과 불일치를 만들어냅니다.

조직으로 치면: 인수인계 노트 없이 교대하는 것과 같습니다.

**해결: 구조화된 인수인계 프로토콜.**

모든 인수인계는 하나의 템플릿을 따릅니다:

```markdown
## Handoff — [timestamp]
> [무엇을 완료했는지 한 줄 요약]

## Completed
- [완료 항목, 파일 경로 포함]

## In Progress
- [부분 완료 항목, 현재 상태 설명]

## Next Steps
- [ ] [구체적이고 실행 가능한 항목]

## Context
- [내린 결정과 그 이유]
- [생성/수정한 파일]
- [발견한 블로커]
```

핵심 섹션은 **Context**입니다 — 무엇을 했는지가 아니라 *왜* 했는지. 의사결정의 근거가 없으면, 받는 쪽 에이전트가 다른 전제를 세우고 작업이 갈라집니다.

인수인계의 *방향*에 대해서도 뼈아픈 교훈을 얻었습니다. 초기에 인수인계는 보내는 쪽 관점("내가 이걸 했다")으로 작성됐습니다. 하지만 *받는 쪽* 관점("이어서 하려면 이것을 알아야 한다")으로 써야 합니다. 차이는 미묘하지만 사용성을 완전히 바꿉니다.

## 충돌 해결: 에이전트끼리 의견이 다를 때

세 에이전트를 운영하면 의견 충돌은 필연입니다. Eve는 문서를 "아카이브 가능"으로 분류하는데 Claude는 "보강이 필요하다"고 봅니다. Chloe의 파이프라인이 항목을 특정 프로젝트로 라우팅하는데, Eve는 그 목적지가 잘못됐다고 판단합니다.

우리는 단순한 위계로 이를 처리합니다:

**1. 공유 상태를 확인한다.** Future Board에 소유권이 지정돼 있으면, 소유자가 결정합니다. 논쟁 없음.

**2. 판단 프레임워크를 확인한다.** 명확한 소유자가 없으면, 양쪽 에이전트가 5축 프레임워크로 의사결정을 스코어링합니다. 높은 점수가 이깁니다 — 근거를 문서화한 상태로.

**3. 사람에게 에스컬레이션한다.** 점수 차이가 0.5점 이내라면(진짜 반반), 사용자에게 플래그합니다. "Eve는 X를 추천합니다 (점수 3.8), Claude는 Y를 추천합니다 (점수 3.6). 결정해 주세요."

실전에서 Step 3은 드물게 발생합니다 (충돌의 ~5%). 대부분의 의견 차이는 Step 1(명확한 소유권)이나 Step 2(스코어링하면 한쪽이 객관적으로 나음)에서 해소됩니다.

## 멀티 에이전트 시스템 5원칙

### 1. 역량이 아니라 환경에 따라 라우팅하라

에이전트가 *할 수 있는 것*을 기준으로 태스크를 배정하지 마세요. 에이전트가 *어디서 돌아가는지*를 기준으로 하세요. 24/7 서버 에이전트가 비동기 작업을, 파이프라인 에이전트가 스케줄 작업을, 인터랙티브 에이전트가 사용자 세션을 처리합니다. 역량이 겹치는 건 괜찮습니다 — 라우팅의 명확성이 중요합니다.

### 2. 맥락이 아니라 상태를 공유하라

컨텍스트 윈도우를 합칠 수는 없습니다. 구조화된 상태는 공유할 수 있습니다: 태스크 보드, JSONL 로그, 공동 헌법. 공유 상태를 설계할 때 하나의 질문에 답하도록 만드세요: "내가 마지막으로 확인한 이후에 뭐가 바뀌었지?"

### 3. 인수인계는 받는 사람을 위해 쓰라

모든 인수인계는 이 질문에 답해야 합니다: "이 태스크에 대해 아무것도 모른다면, 이어서 하려면 뭘 알아야 하지?" 결정의 *이유*를 포함하세요. *무엇*만이 아니라. 파일 경로, 근거, 구체적인 다음 단계 — 요약이 아닌.

### 4. 소유권을 모호하게 두지 마라

두 에이전트가 모두 맡을 수 있는 태스크라면, 정확히 하나가 지정돼야 합니다. 지정 방식은 존재한다는 사실보다 덜 중요합니다. 우리는 단순한 라우팅 테이블을 씁니다. 우선순위 규칙, 라운드 로빈, 역량 매칭 — 어떤 것이든 됩니다. 결정론적이기만 하면.

### 5. 이음새를 설계하라

멀티 에이전트 시스템의 실패 지점은 개별 에이전트 내부가 아닙니다 — 에이전트 *사이의 이음새*입니다. 인수인계 순간. 상태 확인. 충돌 해결. 설계 노력을 개별 퍼포먼스가 아니라 전환 지점에 쏟으세요.

## 멀티 에이전트를 선택할 때 (그리고 하지 말아야 할 때)

멀티 에이전트 시스템은 협업 오버헤드를 추가합니다. 그 오버헤드가 가치를 가지려면:

| 조건 | 단일 에이전트 | 멀티 에이전트 |
|------|-------------|-------------|
| 모든 작업이 인터랙티브 | 충분 | 과잉 |
| 24/7 비동기 처리 필요 | 부족 | 필수 |
| 작업이 여러 환경에 걸쳐 있음 | 어색 | 자연스러운 적합 |
| 단일 세션 용량을 초과하는 볼륨 | 병목 | 확장 가능 |

전체 워크플로우가 터미널 하나에서 벌어진다면, 에이전트 하나가 셋보다 낫습니다. 멀티 에이전트의 협업 비용이 보상받으려면 *환경 다양성*(다른 런타임, 다른 스케줄)이나 단일 세션을 넘어서는 *처리량*이 필요합니다.

## 핵심 요약

멀티 에이전트 시스템은 이음새에서 실패합니다: 소유권 모호, 맥락 단절, 인수인계 누락. 해결책은 더 똑똑한 에이전트가 아닙니다 — 더 나은 **조직 인프라**입니다: 라우팅 테이블, 공유 상태, 구조화된 인수인계, 공동 헌법.

```
Ch1: 설계 능력 > 코딩 능력
Ch2: CLAUDE.md = 헌법
Ch3: 스킬 = 직무기술서
Ch4: 판단 프레임워크 = 의사결정 정책
Ch5: 멀티 에이전트 운영 = 팀 협업 (라우팅, 공유 상태, 인수인계)
```

시스템이 거의 완성됐습니다. 한 가지 도전이 남아 있습니다: 166개의 에셋을 관리하고, 버전을 매기고, 때로는 폐기해야 할 때 무슨 일이 벌어지는가? 그것이 확장의 문제입니다.
