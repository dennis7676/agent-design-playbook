# Chapter 3: 스킬 아키텍처 — 좋은 스킬과 나쁜 스킬을 가르는 기준

> 잘 설계된 스킬은 **직무기술서**이지, 할 일 목록이 아닙니다. 역할을 정의하는 것이지, 작업을 나열하는 게 아닙니다.

## TL;DR

크기가 품질을 결정하지 않습니다. 274줄짜리 스킬이 중복을 이유로 아카이브되었고, 8,000토큰짜리 스킬은 모든 토큰이 존재 이유가 있어서 지금도 잘 돌아가고 있습니다. 이 챕터에서는 실제 프로덕션에서 살아남은 스킬과 사라진 스킬을 비교 분석하여, 일상 운영에서 정말 버텨주는 스킬을 만들기 위한 다섯 가지 설계 원칙을 도출합니다.

---

## 범위의 역설

놀라웠던 경험이 하나 있습니다.

제가 처음으로 아카이브한 스킬은 274줄이었습니다. 권장 상한인 500줄보다 한참 짧죠. 깔끔하게 정리되어 있었고, 주제를 빠짐없이 다루고 있었습니다. 그런데 완전히 불필요했습니다. 다른 스킬이 이미 그 역할의 95%를 처리하고 있었거든요.

반면, 가장 안정적으로 동작하는 스킬 중 하나는 8,000토큰에 달합니다. 어떤 "베스트 프랙티스" 가이드라인보다도 훨씬 큽니다. 하지만 모든 토큰이 구조를 떠받치고 있습니다. 어느 섹션이든 빼면 리서치 파이프라인이 깨집니다.

**교훈**: 스킬의 품질은 크기로 결정되지 않습니다. 핵심은 *정당화 밀도(justification density)* — 모든 줄이 자기 자리를 벌어들이고 있는가입니다. 중복으로 가득 찬 날씬한 스킬이, 필수 내용으로 가득 찬 무거운 스킬보다 더 나쁩니다.

조직설계 비유로 돌아가 보겠습니다. 좋은 직무기술서는 짧지 않습니다. *정확*합니다. 이 역할이 정확히 무엇을 하고, 무엇을 하지 않으며, 다른 역할과 어떻게 연결되는지를 명시합니다. 나쁜 직무기술서는 너무 모호해서 세 사람이 같은 일을 자기 소관이라 착각하거나, 너무 과적되어 한 사람이 네 가지 업무를 하게 만듭니다.

## 실패한 스킬의 해부

실패가 어떻게 생겼는지, 실제 사례로 보여드리겠습니다.

### 사례 분석: 흡수당한 스킬

`n8n-code-nodes`라는 스킬이 있었습니다. n8n Code 노드에서 JavaScript와 Python을 작성하는 방법을 에이전트에게 가르치는 역할이었죠. 구성도 잘 되어 있었습니다 — 에러 패턴, 자주 쓰는 레시피, 언어 비교 테이블. 손닿는 곳에 두고 싶은 종류의 레퍼런스 자료였습니다.

문제는 `n8n-code-and-expressions`라는 스킬도 있었다는 겁니다. 같은 영역을 다루면서 표현식 문법, 빌트인 함수, 디버깅 패턴까지 추가로 커버하고 있었습니다. 겹침 비율 약 95%.

**무엇이 잘못되었나**:

| 이슈 | 증상 |
|------|------|
| **범위 겹침** | 같은 질문에 두 스킬이 경쟁적으로 응답 |
| **모호한 트리거** | "코드 작성 시 OR 에러 해결 시 OR 언어 선택 시 사용" — 세 가지 맥락이 하나의 OR 체인에 뭉쳐 있음 |
| **경계 부재** | "NOT WHEN" 섹션 없음. 다른 스킬로의 명확한 핸드오프도 없음 |
| **필요성 부재** | 제거해도 아무것도 안 깨짐. 다른 스킬이 전부 처리함 |

아카이브된 파일에는 이렇게 적혀 있습니다: *"JS/Python 통합 가이드가 n8n-code-and-expressions에 완전 흡수됨. 고유 콘텐츠 없음."*

조직으로 비유하면, 같은 직무기술서를 가진 두 명의 직원이 있었고, 한 명이 떠나야 했습니다.

### 비대화 패턴

반대쪽 실패는 너무 많은 일을 하는 스킬입니다. 제 스킬 중 하나는 관련 책임을 흡수하면서 6,700토큰까지 불어났습니다 — 유효성 검증 로직, 인물 매칭, 회의록 보강, 위키 등록까지 전부 하나의 파일에.

동작은 했습니다. 하지만 유지보수가 악몽이었습니다. 인물 매칭 로직을 바꾸려면 6,700토큰 분량의 컨텍스트를 통째로 로드해야 했고, 대부분은 관계없는 내용이었습니다. 위키 등록 버그를 고치려면 전체 유효성 검증 파이프라인을 이해해야 했습니다.

평가 리포트에서는 집중된 개별 스킬로 분리할 것을 권고했습니다. 핵심만 남기면 7,200토큰에서 약 3,000토큰으로 줄어들 것이었습니다.

조직으로 비유하면, 한 사람이 네 가지 업무를 하고 있는 상태. 생산적인가요? 기술적으로는 그렇습니다. 지속 가능한가요? 아닙니다.

## 성공한 스킬의 해부

이번에는 성공이 어떻게 생겼는지 보겠습니다.

### 사례 1: 규칙집 (274줄, 3,500토큰)

`obsidian-writing-rules` 스킬은 볼트 전체에 적용되는 서식 규칙 모음입니다: frontmatter 스키마, 태스크 형식, 파일명 규칙, 캡처 템플릿, 일관성 체크. 일곱 개 섹션이 각각 독립적으로 동작합니다.

**왜 잘 동작하는가**:

**1. 트리거가 구문적으로 정확합니다.**

이런 식이 아닙니다: "Obsidian 작업 시 사용." 이러면 모든 것에 매칭됩니다.
대신 이렇습니다: "Obsidian 볼트의 파일을 작성하거나 수정할 때 사용." 조건 하나, 동작 하나, 모호함 제로.

**2. 경계가 정성스럽게 문서화되어 있습니다.**

이 스킬에서 가장 중요한 문단은 gotcha(함정) 항목입니다:

> *사용자가 URL과 함께 "이거 메모 좀 해줘"라고 보내면, 메모로 분류하기 쉽습니다. 하지만 URL이 있으면 사용자가 뭐라고 말했든 상관없이 반드시 URL 캡처로 분류해야 합니다.*

이 문단 하나가 반복적으로 발생하던 분류 오류를 통째로 방지합니다. 일반 원칙이 아닙니다 — 계속 발생했기 때문에 기록해둔, 구체적이고 실전에서 검증된 교정 사항입니다.

**3. 각 섹션이 독립적으로 유용합니다.**

C섹션(파일명 규칙)을 읽기 위해 F섹션(지식 그래프 체크)을 이해할 필요가 없습니다. 에이전트는 전체 스킬을 로드하되 관련 섹션만 활성화합니다. 컨텍스트 낭비가 없습니다.

**테스트 결과**: 구조화된 시나리오 전반에서 83% 통과율. 나머지 17%는 의도적으로 설계된 모호한 엣지 케이스로, 사람의 판단이 필요한 경우입니다 — 스킬은 이를 정확히 그렇게 식별합니다.

### 사례 2: 리서치 엔진 (8,000토큰)

`deep-research` 스킬은 멀티 에이전트 리서치를 오케스트레이션합니다: Haiku급 수집 에이전트를 병렬로 파견하고, Sonnet으로 교차 검증하고, Opus급 추론으로 종합합니다. 여섯 단계, 각각 명확히 분리되어 있습니다.

**크기에도 불구하고 왜 잘 동작하는가**:

**1. 토큰 지출이 모델 비용에 매핑됩니다.**

이 스킬의 핵심 통찰: *"수집은 싸게, 판단은 비싸게(Collect cheap, judge expensive)."* 빠르고 저렴한 에이전트는 수집에 할당하고, 비싼 추론은 종합에 남겨둡니다. 8,000토큰은 이 할당 전략을 가르칩니다 — 체크리스트 수준으로 압축할 수 없는 내용입니다.

**2. 명시적 "NOT WHEN" 섹션이 있습니다.**

> 단순 팩트 질문 → `web-research` 스킬 사용
> 단순 URL 읽기 → `firecrawl` 직접 사용
> 심층 비교 분석, 도구 평가 → 이 스킬 사용

이것이 과도한 트리거 발동을 방지합니다. 이게 없으면 모든 웹 검색마다 8,000토큰의 리서치 파이프라인이 로드될 것입니다.

**3. 실패 한도가 내장되어 있습니다.**

Step 3.5에 서킷 브레이커가 포함되어 있습니다: 최대 라운드 후에도 커버리지가 임계값 미만이면 그냥 진행하되 빈 부분을 "미검증"으로 표시합니다. 이것이 없으면 스킬이 완벽한 커버리지를 달성하려다 무한 루프에 빠질 수 있습니다.

## 스킬 아키텍처의 다섯 가지 원칙

20개 이상의 프로덕션 스킬 — 살아남은 것과 사라진 것 — 으로부터 도출한 원칙입니다:

### 1. 필요성 테스트

스킬을 만들기 전에 물어보세요: *"이 스킬이 내일 사라지면 뭐가 깨지는가?"*

| 답변 | 조치 |
|------|------|
| "아무것도 안 깨짐, 다른 스킬이 처리함" | 만들지 마세요 |
| "X와 Y 작업의 일관성이 무너짐" | 만드세요 |
| "매번 Z를 수동으로 해야 함" | 만드세요 |

아카이브된 `n8n-code-nodes`는 첫날부터 이 테스트에 떨어졌을 겁니다. 현역인 `obsidian-writing-rules`는 결정적으로 통과합니다 — 이것 없이는 볼트에 쓸 때마다 frontmatter가 들쭉날쭉해집니다.

### 2. 트리거는 넓이보다 정밀도

**나쁜 예**: `"코드 작성 시 OR 에러 해결 시 OR 모드 선택 시 사용"` — 세 가지 맥락이 하나의 트리거에 뭉쳐 있습니다.

**좋은 예**: `"[단일 조건] 시 사용"` — 정확히 하나의 시나리오에서 활성화되고, 인접 시나리오는 명시적으로 다른 곳으로 안내합니다.

패턴은 이렇습니다: 좋은 트리거는 `if-then` 문처럼 읽힙니다. 나쁜 트리거는 `if-or-or-or-then` 문처럼 읽힙니다.

### 3. Gotcha(함정 문서)가 구조를 떠받친다

스킬 간 토큰 배분을 비교하면서 발견한 반직관적 사실이 있습니다:

| 스킬 | 예시에 할당된 줄 수 | gotcha에 할당된 줄 수 | 상태 |
|------|-------------------|---------------------|------|
| n8n-code-nodes (아카이브됨) | ~190 | <10 | 사망 |
| obsidian-writing-rules | ~80 | ~100 | 활발 |
| deep-research | ~120 | ~80 | 활발 |

**성공한 스킬은 "정상 동작 예시"보다 "뭐가 잘못될 수 있는가"에 더 많은 토큰을 씁니다.**

각 gotcha는 네 가지 질문에 답해야 합니다:
1. 어떤 구체적 실수를 방지하는가?
2. 왜 미묘한가 (왜 쉽게 눈에 안 띄는가)?
3. 수정 방법은?
4. 이 gotcha가 적용되지 않는 경우는? (범위 한정)

### 4. 출력 계약은 테스트 가능해야 한다

"요약을 생성하라"와 "이 5개 필수 필드를 가진 YAML frontmatter 블록을 생성하라"의 차이는 테스트 가능성입니다.

좋은 스킬은 기계적으로 검증할 수 있는 출력을 정의합니다:
- **obsidian-writing-rules**: frontmatter에 `type`, `created`, `updated`, `status`, `tags`가 반드시 있어야 함 → 기계 검증 가능
- **deep-research**: Step 3에서 커버리지 점수 ≥ 임계값 생성 → 수치 검증 가능
- **skill-quality-gate**: 5개 항목 체크리스트, 각각 통과/실패 → 이진 결과

나쁜 스킬은 모호한 출력을 약속합니다: "도움이 되는 코드," "명확한 설명," "적절한 응답." 테스트할 수 없으니, 실패가 보이지 않습니다.

### 5. 변경 이력은 생존 징후다

버전 이력이 없는 스킬은 완벽하거나(가능성 낮음) 방치되었거나(가능성 높음) 둘 중 하나입니다.

활발하고 건강한 스킬은 다음을 보여줍니다:
- 버전 번호 또는 날짜별 변경 로그 항목
- 반복 개선의 흔적 (섹션 추가, 규칙 정교화)
- 과다/과소 발동 사례를 바탕으로 한 트리거 보정

다른 사람의 스킬(또는 석 달 전 자기 자신의 스킬)을 평가할 때, 변경 이력부터 확인하세요. 마지막 업데이트가 몇 주 전이고 시스템은 계속 진화했다면, 그 스킬은 아마 정렬에서 벗어나고 있을 겁니다.

## 직무기술서 테스트

실용적 연습을 하나 소개합니다. 아무 스킬이나 골라서 "직무기술서 테스트"를 적용해 보세요:

1. **역할**: 이 스킬이 하는 일을 한 문장으로 설명할 수 있는가?
2. **책임 범위**: 경계가 명확한가? 다른 스킬과 겹치지 않는가?
3. **성공 기준**: 스킬이 올바르게 수행됐는지 어떻게 아는가?
4. **에스컬레이션 경로**: 스킬의 범위 밖 상황을 만나면 어떻게 되는가?
5. **온보딩**: 새 에이전트(또는 빈 컨텍스트 윈도우)가 기술서만 보고 이 스킬을 효과적으로 사용할 수 있는가?

어느 하나라도 "불명확" 또는 "상황에 따라 다름"이 답이라면, 그 스킬은 재설계가 필요합니다.

조직에서 모호한 직무기술서는 영역 다툼, 빠뜨린 업무, 중복 작업을 의미합니다. 에이전트 시스템에서 모호한 스킬은 똑같은 것을 의미합니다 — 다만 잘못된 트리거 발동, 일관성 없는 출력, 낭비된 토큰이라는 형태로 나타날 뿐입니다.

## 핵심 정리

**좋은 스킬은 작은 것이 아니라 정밀한 것입니다.** 모든 토큰이 필요성으로 자리를 벌어들이고, 경계는 문서화된 실패를 통해 정의되며, "잘되는 경우"보다 "잘못되는 경우"에 더 많은 노력을 투자합니다.

Chapter 1부터 여기까지의 흐름:

```
Ch1: 설계 능력 > 코딩 능력
Ch2: CLAUDE.md = 헌법 (지도 → 가드레일 → 인지)
Ch3: 스킬 = 직무기술서 (정밀한 범위, 명확한 경계, 테스트 가능한 출력)
```
