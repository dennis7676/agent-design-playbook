# 워크시트 5: 멀티 에이전트 시스템 설계

> [5장 — 멀티 에이전트 운영](../chapters/ch05-multi-agent-ops.md) 함께 보기

## 시작하기 전에

이 워크시트는 에이전트가 여러 개 필요한지 판단하고, 필요하다면 어떻게 조율할지 설계하는 데 사용합니다. **모든 사람에게 멀티 에이전트가 필요한 건 아닙니다** — 그래서 그 질문부터 시작해요.

---

## 실습 1: 멀티 에이전트가 정말 필요한가? (5분)

해당하는 항목을 모두 체크하세요:

- [ ] 내가 컴퓨터 앞에 없을 때도 처리가 돌아가야 한다 (비동기)
- [ ] 작업이 여러 환경에 걸쳐 있다 (터미널 + 서버 + 파이프라인)
- [ ] 하루 작업량을 단일 세션으로 감당할 수 없다
- [ ] 작업마다 다른 모델이나 도구가 필요하다

**0~1개 체크**: 멀티 에이전트가 필요 없을 가능성이 높습니다. 잘 설계된 에이전트 하나가 더 낫습니다.
**2개 이상 체크**: 조율 비용을 감수할 만큼 멀티 에이전트가 유의미할 수 있습니다. 계속 진행하세요.

---

## 실습 2: 역할 정의 (10분)

각 에이전트의 역할을 정의하세요. 핵심 원칙: **능력이 아닌 환경 기준으로 나누세요.**

| 에이전트 | 실행 환경 | 가용 시간 | 주요 역할 |
|---------|----------|----------|----------|
| 에이전트 1 | [예: 터미널] | [예: 필요 시] | |
| 에이전트 2 | [예: 서버] | [예: 24/7] | |
| 에이전트 3 | [예: 파이프라인] | [예: 스케줄] | |

**중복 점검**: 에이전트 쌍별로, 둘 다 처리할 수 있는 작업을 나열하세요:

| 에이전트 쌍 | 겹치는 작업 | 담당자는? | 이유 |
|------------|-----------|----------|------|
| 1 & 2 | | | |
| 1 & 3 | | | |
| 2 & 3 | | | |

겹치는 작업은 반드시 담당자가 한 명이어야 합니다.

## 실습 5: 인수인계 템플릿 작성 (10분)

에이전트 간 인수인계에 사용할 형식을 설계하세요:

```markdown
## 인수인계 — [타임스탬프]
> [한 줄 요약]

## 완료
- [끝난 작업]

## 진행 중
- [부분 완료 + 현재 상태]

## 다음 단계
- [ ] [받는 에이전트가 할 구체적 행동 항목]

## 맥락
- [내린 결정과 그 이유]
- [수정한 파일]
- [차단 요소]
```

**검증**: 사전 맥락이 전혀 없는 에이전트가 이 인수인계만 읽고 올바르게 작업을 이어갈 수 있나요?

## 실습 6: 충돌 해결 프로토콜 (5분)

두 에이전트의 판단이 충돌하면 어떻게 하나요?

1단계: _________________________________ (예: 담당자 확인)
2단계: _________________________________ (예: 판단 프레임워크로 점수 매기기)
3단계: _________________________________ (예: 사람에게 에스컬레이션)

충돌 중 사람까지 올라가는 비율은 몇 %가 적정한가요? 목표: 10% 미만.

---

## 토론 주제 (스터디 그룹)

1. **"멀티 에이전트를 도입했다가 후회한 사람?"** 조율 비용이 얼마나 들었나요? 그만한 가치가 있었나요?

2. **"최악의 인수인계 실패 경험은?"** 어떤 정보가 유실됐나요? 템플릿을 어떻게 고쳤나요?

3. **"단일 에이전트 vs. 멀티 에이전트 — 상대편을 설득해 보세요."** 트레이드오프를 토론해 봅시다.

## 산출물

이 워크시트를 마치면 아래 항목이 준비되어야 합니다:
- [ ] 멀티 에이전트가 필요한지에 대한 명확한 예/아니오 판단
- [ ] 필요하다면: 역할 테이블, 라우팅 테이블, 공유 상태 설계, 인수인계 템플릿
- [ ] 필요 없다면: 문서화된 근거 (나중에 재검토할 때 유용합니다)
